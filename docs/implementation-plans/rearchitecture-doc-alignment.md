# リアーキテクチャ方針に沿った主要ドキュメント再整理計画

## 概要

`docs/architecture/re-architecture.md` を基準文書として、リポジトリ直下の `README.md` と `AGENTS.md` を再整理する。

今回の目的は、旧来の説明で混在している「現行の実装構成」と「これから目指す再設計方針」を切り分け、開発者と AI エージェントの両方が同じ前提で作業できる状態にすることである。

## 背景

現行の `README.md` と `AGENTS.md` には、以下の混在がある。

- 現在の workspace 構成
- 将来的に目指す構成
- 旧アーキテクチャ前提の説明
- AI 作業ルールとプロジェクト説明の混在

`docs/architecture/re-architecture.md` では、プロジェクトを「Rust 製の静的コンテンツ公開基盤 + SSR 表示基盤」として再定義しているため、主要ドキュメントもその認識に合わせて整理する必要がある。

## この改訂で揃える方針

### 1. 現状と目標を分離して書く

- 現在存在する crate / app と、今後目指す構成を明確に分ける
- 未作成の `application` / `infrastructure` / `apps/server` / `apps/publisher` を、既存構成としては記述しない
- 将来像は「移行先」「目標構成」として示す

### 2. パイプライン中心の説明に寄せる

- プロジェクトの主役を常駐 API サーバーではなく、公開成果物生成パイプラインとして説明する
- `obsidian_uploader` を、将来的な `publisher` 的責務を持つアプリとして位置付ける
- Leptos SSR は、公開成果物を読む表示基盤として説明する

### 3. 実運用に沿った責務分割を明示する

- Markdown から HTML への変換はビルド時に行う
- ランタイムのサーバーは S3 上の成果物を読む
- VPS + systemd + nginx + 単一バイナリ運用を前提として整理する

### 4. AI 向け指示は AGENTS.md に寄せて整理する

- 優先して参照すべき文書の順序を定義する
- 実装前に `docs/implementation-plans/` を追加するルールを維持する
- `terraform/` の制約、TDD、ドメイン純粋性などの必須ルールを簡潔に残す

## 各ドキュメントの責務

### `README.md`

- 最終的な目標像の説明
- リポジトリ全体の責務分割
- データフローと運用方針
- 実際に利用できる開発コマンド

### `AGENTS.md`

- AI エージェント向けの作業ルール
- 優先参照ドキュメント
- アーキテクチャ上の禁止事項と非目標
- 文書更新時の注意点

## 改訂時の注意

- 実在しない構成を、現在の実装として断定しない
- `README.md` では利用可能な `cargo make` タスクのみを記載する
- `docs/adr/` は参考情報として残しつつ、再設計方針との優先順位を明記する
- 具体的なコード例よりも、責務分割と依存方向を正しく伝えることを優先する

## 完了条件

- `README.md` が、現状と移行方針を混同せずに読める
- `AGENTS.md` が、AI 作業ルールとして過不足なく機能する
- `docs/architecture/re-architecture.md` を最上位の再設計メモとして参照できる導線がある
- 主要ドキュメント間で、プロジェクトの目的と非目標に矛盾がない
